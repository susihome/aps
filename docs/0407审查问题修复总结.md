## 高优先级问题修复

### 1. ✅ 授权表达式 `principal.id` 问题

**问题：** `@PreAuthorize("hasRole('ADMIN') or #id == authentication.principal.id")` 中的 `principal.id` 不存在

**根本原因：**
- `JwtAuthenticationFilter` 将 `username` 字符串作为 principal
- `CustomUserDetailsService` 返回标准 `UserDetails`，没有 `id` 字段

**解决方案：**

#### 创建 UserPrincipal
```java
@Getter
public class UserPrincipal extends User {
    private final UUID id;
    
    public UserPrincipal(UUID id, String username, String password,
                         Collection<? extends GrantedAuthority> authorities) {
        super(username, password, authorities);
        this.id = id;
    }
}
```

#### 更新 CustomUserDetailsService
```java
@Override
public UserDetails loadUserByUsername(String username) {
    User user = userRepository.findByUsername(username)
            .orElseThrow(() -> new UsernameNotFoundException("用户不存在: " + username));
    
    return new UserPrincipal(
            user.getId(),
            user.getUsername(),
            user.getPasswordHash(),
            user.getEnabled(),
            true, true, true,
            getAuthorities(user)
    );
}
```

#### 更新 JwtAuthenticationFilter
```java
UserPrincipal principal = new UserPrincipal(
        userId,
        username,
        "",
        authorities
);

UsernamePasswordAuthenticationToken authentication =
        new UsernamePasswordAuthenticationToken(principal, null, authorities);
```

**验证：**
- `authentication.principal.id` 现在可以正确访问 UUID
- `@PreAuthorize` 表达式不会抛出 `SpelEvaluationException`

### 4. ✅ 统一 API 返回结构

**问题：** `UserController` 和 `OrderController` 返回 `ResponseEntity`，与 `AuthController` 的 `AjaxResult` 不一致

**解决方案：**

#### UserController（修改前）
```java
@GetMapping
public ResponseEntity<List<UserDto>> getAllUsers() {
    return ResponseEntity.ok(userDtos);
}
```

#### UserController（修改后）
```java
@GetMapping
public AjaxResult<List<UserDto>> getAllUsers() {
    return AjaxResult.success(userDtos);

```

- API 契约清晰

## 中优先级问题修复

### 5. ✅ AuditAspect 提取真实 userId

**问题：** `UUID userId = null; // 可以从 authentication 中提取` - 审计日志用户 ID 字段长期为空

**解决方案：**

```java
// 获取当前用户信息
Authentication authentication = SecurityContextHolder.getContext().getAuthentication();
String username = authentication != null ? authentication.getName() : "anonymous";
UUID userId = null;

// 从 UserPrincipal 提取 userId
if (authentication != null && authentication.getPrincipal() instanceof UserPrincipal) {
    UserPrincipal principal = (UserPrincipal) authentication.getPrincipal();
    userId = principal.getId();
}
```

**验证：**
- 审计日志现在正确记录 `user_id`
- 可追溯性完整

### 6. ✅ Cookie 安全配置化

**问题：** `cookie.setSecure(false)` 硬编码，生产环境不安全

**解决方案：**

#### application.yml
```yaml
jwt:
  cookie:
    secure: ${JWT_COOKIE_SECURE:false}  # 生产环境设为 true
```

#### AuthController
```java
@Value("${jwt.cookie.secure:false}")
private boolean cookieSecure;

private void setAccessTokenCookie(HttpServletResponse response, String accessToken) {
    Cookie cookie = new Cookie("accessToken", accessToken);
    cookie.setHttpOnly(true);
    cookie.setSecure(cookieSecure);  // ✅ 配置驱动
    cookie.setPath("/");
    cookie.setMaxAge((int) (jwtExpiration / 1000));
    response.addCookie(cookie);
}
```

**生产环境配置：**
```bash
export JWT_COOKIE_SECURE=true
```

## 关键教训

### 1. 授权表达式需要匹配实际 Principal 类型
```java
// ❌ 错误：假设 principal 有 id 字段
@PreAuthorize("#id == authentication.principal.id")

// ✅ 正确：自定义 Principal 携带 id
public class UserPrincipal extends User {
    private final UUID id;
}
```

### 2. 安全配置不要硬编码
```java
// ❌ 错误
cookie.setSecure(false);

// ✅ 正确
@Value("${jwt.cookie.secure:false}")
private boolean cookieSecure;
cookie.setSecure(cookieSecure);
```

### 3. API 契约要统一
- 成功响应和失败响应使用相同格式
- 避免前端需要按接口分支解析
