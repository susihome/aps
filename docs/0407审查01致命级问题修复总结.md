### 2. ✅ DB Schema 与 JPA Entity 列名不匹配

**问题：**
- `ddl-auto: validate` 会严格校验表结构
- 3 处列名不匹配：
  - `users.password` → 应该是 `password_hash`
  - `permissions.name` → 应该是 `code`
  - `schedules.score` → 应该是 `final_score`

**解决方案：**

#### V1__Initial_Schema.sql 修改
```sql
-- 修改前
CREATE TABLE users (
    password VARCHAR(255) NOT NULL,
    ...
);

-- 修改后
CREATE TABLE users (
    password_hash VARCHAR(255) NOT NULL,
    ...
);

-- 修改前
CREATE TABLE permissions (
    name VARCHAR(100) UNIQUE NOT NULL,
    ...
);

-- 修改后
CREATE TABLE permissions (
    code VARCHAR(100) UNIQUE NOT NULL,
    ...
);

-- 修改前
CREATE TABLE schedules (
    score VARCHAR(100),
    ...
);

-- 修改后
CREATE TABLE schedules (
    final_score VARCHAR(100),
    ...
);
```

**验证：**
- 列名与 JPA Entity 映射完全匹配
- `ddl-auto: validate` 不会报错

### 3. ✅ CSRF 阻断所有认证后的写操作

**问题：**
- 只有 `/api/auth/**` 和 `/ws/**` 豁免 CSRF
- 会话策略是 `STATELESS`（无 Session）
- CSRF token 基于 Session 存储
- 所有认证后的 POST/PUT/DELETE 请求都会被 CSRF 拦截返回 403

**解决方案：**

#### SecurityConfig.java
```java
// 修改前
.csrf(csrf -> csrf
    .ignoringRequestMatchers("/api/auth/**", "/ws/**")
)

// 修改后
.csrf(csrf -> csrf.disable())  // Cookie-based JWT + STATELESS 架构禁用 CSRF，依赖 SameSite cookie
```

**安全保障：**
- Cookie 已配置 `HttpOnly=true`
- Cookie 已配置 `SameSite=Lax`（可配置为 Strict）
- Cookie `secure` 属性配置化（生产环境设为 true）

### 4. ✅ Solver endTime 永远不会被重算

**问题：**
- `startTime` 是 `@PlanningVariable`，Solver 会修改它
- `endTime` 不是 `@ShadowVariable`，没有 listener 重算
- Solver 改了 `startTime` 后，`endTime` 仍是旧值
- 所有依赖 `endTime` 的约束（`resourceConflict`、`operationSequence`、`minimizeDelay`）全部在用过期数据评估
- **求解结果无意义**

**解决方案：**

#### AssignmentPlanningModel.java
```java
// 添加 @ShadowVariable 注解
@ShadowVariable(
        variableListenerClass = EndTimeUpdatingVariableListener.class,
        sourceVariableName = "startTime"
)
private LocalDateTime endTime;

// 添加 Variable Listener
public static class EndTimeUpdatingVariableListener implements VariableListener<SchedulePlanningModel, AssignmentPlanningModel> {
    
    @Override
    public void afterVariableChanged(ScoreDirector<SchedulePlanningModel> scoreDirector, AssignmentPlanningModel assignment) {
        updateEndTime(scoreDirector, assignment);
    }
    
    private void updateEndTime(ScoreDirector<SchedulePlanningModel> scoreDirector, AssignmentPlanningModel assignment) {
        LocalDateTime startTime = assignment.getStartTime();
        Operation operation = assignment.getOperation();

        LocalDateTime endTime;
        if (startTime == null || operation == null || operation.getStandardDuration() == null) {
            endTime = null;
        } else {
            endTime = startTime.plusMinutes(operation.getStandardDuration());
        }

        scoreDirector.beforeVariableChanged(assignment, "endTime");
        assignment.setEndTime(endTime);
        scoreDirector.afterVariableChanged(assignment, "endTime");
    }
}
```

**关键改进：**
- `endTime` 现在是 `@ShadowVariable`
- 当 `startTime` 变化时，`EndTimeUpdatingVariableListener` 自动触发
- `endTime = startTime + operation.standardDuration`
- 约束评估使用正确的 `endTime` 值

## 关键教训

### 1. Flyway 迁移脚本管理
- 不要创建重复的迁移脚本
- 列名必须与 JPA Entity 映射一致
- 使用 `ddl-auto: validate` 严格校验

### 2. CSRF 与 STATELESS 会话
- Cookie-based JWT + STATELESS 架构应禁用 CSRF
- 依赖 `HttpOnly` + `SameSite` cookie 保护
- 或使用 `CookieCsrfTokenRepository`

### 3. Timefold Shadow Variables
- Planning variables 变化时，依赖字段必须用 `@ShadowVariable` 自动更新
- 不能手动计算，必须通过 `VariableListener`
- 否则约束评估使用过期数据
