## 高优先级问题修复（运行期 Bug）

### 1. ✅ operationSequence 约束 Joiner NPE

**问题：**
- 左侧已被 `.filter()` 保护，但右侧（被 join 的 entity）没有做 null 保护
- 如果任何 AssignmentPlanningModel 的 `operation` 或 `operation.order` 为 null，Joiner lambda 会 NPE

**解决方案：**

#### ApsConstraintProvider.java (Line 50-53)
```java
// 修改前
Joiners.equal(
    a -> a.getOperation().getOrder().getId(),
    a -> a.getOperation().getOrder().getId()
),

// 修改后
Joiners.equal(
    a -> a.getOperation() != null && a.getOperation().getOrder() != null
            ? a.getOperation().getOrder().getId() : null,
    a -> a.getOperation() != null && a.getOperation().getOrder() != null
            ? a.getOperation().getOrder().getId() : null
),
```

**验证：**
- 右侧 lambda 现在有完整的 null 保护
- 避免 NPE 导致求解崩溃

### 2. ✅ minimizeDelay 约束遗漏 priority null 检查

**问题：**
- filter 检查了 `getDueDate() != null`，但没有检查 `getPriority() != null`
- 如果 order 没有设优先级 → NPE

**解决方案：**

#### ApsConstraintProvider.java (Line 68-73)
```java
// 修改前
.filter(a -> a.getEndTime() != null
        && a.getOperation() != null
        && a.getOperation().getOrder() != null
        && a.getOperation().getOrder().getDueDate() != null
        && a.getEndTime().isAfter(a.getOperation().getOrder().getDueDate()))

// 修改后
.filter(a -> a.getEndTime() != null
        && a.getOperation() != null
        && a.getOperation().getOrder() != null
        && a.getOperation().getOrder().getDueDate() != null
        && a.getOperation().getOrder().getPriority() != null
        && a.getEndTime().isAfter(a.getOperation().getOrder().getDueDate()))
```

**验证：**
- 添加了 `getPriority() != null` 检查
- 避免 NPE 导致求解崩溃

### 3. ✅ UserService/OrderService 使用 RuntimeException

**问题：**
- `RuntimeException` 不会被 GlobalExceptionHandler 的 BusinessException handler 捕获
- 会落入通用 `Exception` handler，一律返回 HTTP 500
- "用户名已存在"应该是 409，"用户不存在"应该是 404

**解决方案：**

#### 创建专用异常类

**ResourceConflictException.java** (HTTP 409)
```java
public class ResourceConflictException extends BusinessException {
    public ResourceConflictException(String message) {
        super(409, message);
    }
}
```

**ResourceNotFoundException.java** (HTTP 404)
```java
public class ResourceNotFoundException extends BusinessException {
    public ResourceNotFoundException(String message) {
        super(404, message);
    }
}
```

**UnauthorizedException.java** (HTTP 401)
```java
public class UnauthorizedException extends BusinessException {
    public UnauthorizedException(String message) {
        super(401, message);
    }
}
```

#### UserService.java 修改
```java
// 修改前
throw new RuntimeException("用户名已存在: " + username);
throw new RuntimeException("用户不存在: " + userId);
throw new RuntimeException("角色不存在: " + roleType);
throw new RuntimeException("原密码不正确");

// 修改后
throw new ResourceConflictException("用户名已存在: " + username);
throw new ResourceNotFoundException("用户不存在: " + userId);
throw new ResourceNotFoundException("角色不存在: " + roleType);
throw new UnauthorizedException("原密码不正确");
```

#### OrderService.java 修改
```java
// 修改前
throw new RuntimeException("Order not found");

// 修改后
throw new ResourceNotFoundException("Order not found");
```

**验证：**
- 异常现在返回正确的 HTTP 状态码
- 前端可以区分不同类型的错误

### 4. ✅ ScheduleService.solveAsync 竞态条件

**问题：**
- `containsKey` 和 `get` 之间非原子操作
- 并发请求下，另一个线程可能在这两行之间执行了 `remove(scheduleId)`
- 导致 `get()` 返回 null → NPE

**解决方案：**

#### ScheduleService.java (Line 52-58)
```java
// 修改前
if (activeSolverJobs.containsKey(scheduleId)) {
    SolverStatus status = activeSolverJobs.get(scheduleId).getSolverStatus();
    if (status == SolverStatus.SOLVING_ACTIVE) {
        throw new BusinessException("排产方案正在求解中，请勿重复提交");
    }
}

// 修改后
// 检查是否已有求解任务在运行（原子操作）
SolverJob<SchedulePlanningModel, UUID> existingJob = activeSolverJobs.get(scheduleId);
if (existingJob != null && existingJob.getSolverStatus() == SolverStatus.SOLVING_ACTIVE) {
    throw new BusinessException("排产方案正在求解中，请勿重复提交");
}
```

**验证：**
- 使用单次 `get()` 调用，避免竞态条件
- 先检查 null，再调用方法

### 5. ✅ ScheduleService.solveAsync 操作脱管实体

**问题：**
- `schedule` 在 `solveAsync` 方法的事务中加载
- `CompletableFuture.runAsync` 的回调在另一个线程、事务已结束后执行
- `schedule` 已脱管，其懒加载关联（如 `assignments`）此时访问会抛 `LazyInitializationException`

**解决方案：**

#### ScheduleService.java (Line 73-97)
```java
// 修改前
CompletableFuture.runAsync(() -> {
    try {
        SchedulePlanningModel finalBestSolution = solverJob.getFinalBestSolution();
        String finalScore = finalBestSolution.getScore() != null
            ? finalBestSolution.getScore().toString()
            : "N/A";

        log.info("求解完成 - Schedule: {}, Final Score: {}", scheduleId, finalScore);

        // 更新 Schedule Entity
        modelConverter.updateScheduleFromModel(schedule, finalBestSolution);

        // 保存最终结果
        saveSchedule(schedule);
        activeSolverJobs.remove(scheduleId);

        // 发布求解完成事件
        eventPublisher.publishEvent(new SolveCompletedEvent(scheduleId, finalScore));
    } catch (Exception e) {
        log.error("求解失败或保存结果失败", e);
        activeSolverJobs.remove(scheduleId);
        eventPublisher.publishEvent(new SolveFailedEvent(scheduleId, e.getMessage()));
    }
});

// 修改后
CompletableFuture.runAsync(() -> {
    try {
        SchedulePlanningModel finalBestSolution = solverJob.getFinalBestSolution();
        String finalScore = finalBestSolution.getScore() != null
            ? finalBestSolution.getScore().toString()
            : "N/A";

        log.info("求解完成 - Schedule: {}, Final Score: {}", scheduleId, finalScore);

        // 在新事务中重新加载 Schedule 并更新
        updateScheduleWithSolution(scheduleId, finalBestSolution, finalScore);
        activeSolverJobs.remove(scheduleId);

        // 发布求解完成事件
        eventPublisher.publishEvent(new SolveCompletedEvent(scheduleId, finalScore));
    } catch (Exception e) {
        log.error("求解失败或保存结果失败", e);
        activeSolverJobs.remove(scheduleId);
        eventPublisher.publishEvent(new SolveFailedEvent(scheduleId, e.getMessage()));
    }
});
```

#### 新增方法
```java
/**
 * 在新事务中更新排产结果（避免 LazyInitializationException）
 */
@Transactional
public void updateScheduleWithSolution(UUID scheduleId, SchedulePlanningModel finalBestSolution, String finalScore) {
    Schedule schedule = scheduleRepository.findById(scheduleId)
            .orElseThrow(() -> new ResourceNotFoundException("排产方案不存在"));

    // 更新 Schedule Entity
    modelConverter.updateScheduleFromModel(schedule, finalBestSolution);
    schedule.setFinalScore(finalScore);

    // 保存最终结果
    scheduleRepository.save(schedule);
}
```

**关键改进：**
- 异步回调中不再使用脱管的 `schedule` 实体
- 在新事务中重新加载 `schedule`，确保在事务上下文中操作
- 避免 `LazyInitializationException`

## 关键教训

### 1. Timefold 约束 Null 安全
- 所有约束规则必须完整的 null 检查
- Joiner 的左右两侧都需要保护
- filter 必须检查所有可能为 null 的字段

### 2. 异常类型与 HTTP 状态码
- 使用专用异常类（ResourceNotFoundException, ResourceConflictException, UnauthorizedException）
- 避免使用通用 RuntimeException
- 让 GlobalExceptionHandler 正确映射 HTTP 状态码

### 3. 并发安全
- ConcurrentHashMap 的 `containsKey` + `get` 不是原子操作
- 使用单次 `get()` 调用，先检查 null
- 或使用 `computeIfAbsent` 等原子方法

### 4. JPA 事务与异步操作
- 异步回调在事务外执行，实体已脱管
- 在新事务中重新加载实体
- 避免在异步回调中访问懒加载关联
