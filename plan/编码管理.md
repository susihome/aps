# 字典管理实施方案

本方案将在“系统管理”下新增“编码管理”功能，用于统一维护系统中使用的各类业务编码，并按现有 APS 前后端架构补齐菜单、页面、接口、实体与权限。

## 现状判断

- 前端左侧“系统管理”菜单来自 `aps-web/src/router/index.ts` 的路由 `meta.group = '系统管理'`，新增路由即可自动展示。
- 前端系统管理现有页面为 `Users.vue`、`Permissions.vue`、`Roles.vue`、`AuditLog.vue`、`FactoryCalendar.vue`，页面风格以独立视图 + `src/api/*.ts` 调用接口为主。
- 后端当前已有 `UserController`、`RoleController`、`PermissionController`、`FactoryCalendarController` 等模块，但未发现现成的“字典管理/数据字典/编码管理”实体或接口。
- 领域层已有多个枚举与编码类概念，但缺少统一、可配置的字典主数据维护入口。

## 建议目标范围

建议把“字典管理”设计成两层结构：

1. 字典类型
   - 维护字典分类，例如：订单状态、优先级、设备状态、班次类型、工序类型等。
   - 关键字段建议包含：`code`、`name`、`description`、`enabled`、`sortOrder`。

2. 字典项
   - 维护某个字典类型下的具体编码项。
   - 关键字段建议包含：`dictTypeCode`、`itemCode`、`itemName`、`itemValue`、`description`、`enabled`、`sortOrder`、`isSystem`。

这样既能覆盖“系统中用到的编码”，又保留后续扩展空间。

本次范围已确认按上述“编码类型 + 编码项”两层结构实施。

## 实施步骤

### 1. 后端新增字典域模型与持久化

在现有分层结构下新增：

- `aps-domain`
  - 字典类型实体
  - 字典项实体
- `aps-service`
  - Repository
  - Service
  - 必要的校验与异常处理
- `aps-api`
  - Controller
  - 请求/响应 DTO

建议提供的核心能力：

- 字典类型分页/列表查询
- 新增字典类型
- 编辑字典类型
- 启用/禁用字典类型
- 删除字典类型（需校验是否存在字典项）
- 根据字典类型查询字典项列表
- 新增字典项
- 编辑字典项
- 启用/禁用字典项
- 删除字典项
- 提供按字典类型编码查询启用字典项的接口，供业务下拉框复用

### 1.1 数据库结构与约束（新增）

建议新增两张表：

1. `sys_dict_type`
   - 字段：`id`、`code`、`name`、`description`、`enabled`、`sort_order`、`created_at`、`updated_at`
   - 唯一约束：`uk_sys_dict_type_code(code)`
   - 索引：`idx_sys_dict_type_enabled_sort(enabled, sort_order)`

2. `sys_dict_item`
   - 字段：`id`、`dict_type_id`、`item_code`、`item_name`、`item_value`、`description`、`enabled`、`sort_order`、`is_system`、`created_at`、`updated_at`
   - 唯一约束：`uk_sys_dict_item_type_code(dict_type_id, item_code)`
   - 索引：`idx_sys_dict_item_type_enabled_sort(dict_type_id, enabled, sort_order)`
   - 外键：`fk_sys_dict_item_type(dict_type_id -> sys_dict_type.id)`
   - 删除策略：`RESTRICT`（存在字典项时禁止删除字典类型）

迁移脚本命名建议：`V11__Dictionary_Management.sql`（按当前 Flyway 版本号顺延调整）。

## 2. 权限与审计接入

权限建议拆分“类型”和“字典项”两组：

- `system:dict:type:list`
- `system:dict:type:add`
- `system:dict:type:edit`
- `system:dict:type:remove`
- `system:dict:type:query`

- `system:dict:item:list`
- `system:dict:item:add`
- `system:dict:item:edit`
- `system:dict:item:remove`
- `system:dict:item:query`

- `system:dict:query`（通用字典查询接口，供业务页面下拉复用）

关键写操作必须接入审计注解：
- 字典类型：新增、编辑、删除、启停
- 字典项：新增、编辑、删除、启停

## 3. 前端新增菜单、路由与 API

建议新增：

- 路由：`/dictionary`
- 菜单分组：`系统管理`
- 菜单标题：`编码管理`
- 图标：优先复用现有 Element Plus 图标，避免额外改造侧边栏图标映射成本

前端 API 层建议新增：

- `aps-web/src/api/dictionary.ts`

建议封装：

- 查询字典类型列表
- 新增/编辑/删除字典类型
- 查询字典项列表
- 新增/编辑/删除字典项
- 启用/禁用操作
- 获取某字典的启用项（供业务页面后续复用）

### 3.1 API 契约建议（新增）

统一返回格式沿用项目现有响应封装。

1. 字典类型
- `GET /api/dictionaries/types`
  - 入参：`pageNo`、`pageSize`、`keyword`、`enabled`
  - 出参：分页列表（按 `sortOrder asc, id asc`）
- `POST /api/dictionaries/types`
- `PUT /api/dictionaries/types/{id}`
- `PATCH /api/dictionaries/types/{id}/enabled`
- `DELETE /api/dictionaries/types/{id}`

2. 字典项
- `GET /api/dictionaries/types/{typeId}/items`
  - 入参：`pageNo`、`pageSize`、`keyword`、`enabled`
  - 出参：分页列表（按 `sortOrder asc, id asc`）
- `POST /api/dictionaries/types/{typeId}/items`
- `PUT /api/dictionaries/items/{id}`
- `PATCH /api/dictionaries/items/{id}/enabled`
- `DELETE /api/dictionaries/items/{id}`

3. 业务复用查询
- `GET /api/dictionaries/{typeCode}/enabled-items`
  - 仅返回启用项，按 `sortOrder asc, id asc`

4. 错误码与幂等
- `DICT_TYPE_CODE_DUPLICATE`：字典类型编码重复
- `DICT_ITEM_CODE_DUPLICATE`：字典项编码重复
- `DICT_TYPE_HAS_ITEMS`：类型下仍有字典项，禁止删除
- 启停接口保持幂等：重复提交相同状态返回成功

## 4. 前端页面交互建议

建议采用“左右分栏”或“上下主从表”模式，贴近当前 `Roles.vue` 的交互习惯：

- 左侧/上方：字典类型列表
- 右侧/下方：当前类型的字典项列表

建议页面能力：

- 字典类型搜索
- 选择某一类型后自动加载字典项
- 类型与字典项分别支持新增、编辑、删除、启停
- 对系统内置字典项可增加只读/禁止删除控制
- 编码字段创建后建议限制修改，避免影响业务引用

## 5. 与现有业务的衔接策略

这次优先先把“编码管理”模块做出来，不强行替换现有 Java `enum`。

本次范围已确认只实现“编码管理”模块本身，暂不联动改造现有业务页面的数据来源。

分两阶段推进更稳妥：

- 第一阶段
  - 先完成编码管理的维护能力与通用查询接口
  - 新业务页面优先消费字典接口
- 第二阶段
  - 逐步评估将部分前后端硬编码枚举迁移到字典表
  - 对仍适合编译期常量的枚举保持不动

这样风险更低，也便于渐进式改造。

## 风险点

- 当前项目里部分“编码”来源于后端枚举，是否要与字典表双向同步，需要后续单独评估。
- 若权限体系使用固定初始化数据，需要同步补入“字典管理”菜单与按钮权限，否则前端即使有页面也可能不可见。
- 若数据库尚无迁移脚本体系，新增表结构时需要确认当前项目采用的建表方式。

## 预置编码

本次需要预置以下编码类型及基础编码项：

- 订单状态
- 优先级
- 设备状态

具体编码值建议优先复用现有后端枚举语义，保持兼容。

### 预置策略（新增）

- `code`：优先与现有枚举常量语义一致（稳定优先）
- `name`：可使用中文展示名
- `itemValue`：与当前业务比较逻辑保持一致
- `isSystem=true`：系统预置项默认不可删除（可按需允许启停）

## 验收标准（新增）

- 菜单“系统管理/编码管理”可见（具备权限时）
- 类型与字典项支持：查询、新增、编辑、删除、启停
- 重复编码校验生效并返回明确业务错误码
- 删除有字典项的类型时被正确拦截
- 通用查询接口可返回某 `typeCode` 的启用项
- 关键写操作均产生日志审计记录
- 预置编码完成初始化并可在页面查看
